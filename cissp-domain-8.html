Databases — a DBMS is built from hardware, software (the database → tables → rows /columns/fields/keys), the SQL language, users and data. Integrity is preserved through concurrency control, locking, and the ACID properties (Atomicity, Consistency, Isolation, Durability). SQL injection is the signature database attack, defeated by parameterised queries and input validation.

Sub-domain 8.1

Understand & integrate security in the SDLC.

Security is not a phase — it is present in every phase of the software and system life cycles. The single most important idea in the whole domain: bake it in, don't bolt it on.

8.1

The Life Cycles — SLC & SDLC

SDLCPlan → Requirements → Design → Dev → Test → Deploy
SLCSDLC + Operation + Disposal
Cheapest fixAt requirements / design time
Golden ruleBake in security — never bolt on

The Software Development Life Cycle (SDLC) is the structured process for building software: plan and management approval, requirements, architecture & design, development, testing, and deployment. The wider System Life Cycle (SLC) wraps the SDLC and adds the post-launch stages — operation (run, monitor, patch) and disposal (secure decommission and data sanitisation). Security activities live in every phase: requirements capture security requirements, design threat-models, development uses secure coding and SAST, testing runs SAST/DAST, and operation patches continuously.

The exam rewards one instinct above all: defects cost an order of magnitude more to fix at each later phase, so the earliest phase always wins. A design flaw caught on the whiteboard is a meeting; the same flaw in production is an incident.

Exam tip: If a stem offers “add a security review at the requirements phase” against any later-phase fix, the earliest phase is the answer — cheapest and most effective.
8.1

Development Methodologies — Waterfall, Agile, DevOps

WaterfallSequential; rigid; can't go back
AgileIterative sprints; flexible
DevOpsContinuous CI/CD; Dev+QA+Ops united
DevSecOpsShift left; automate security

Waterfall is sequential — each phase is signed off before the next begins. Predictable and heavily documented, but rigid: you cannot easily go back, so a missed requirement is expensive to fix late. Best for stable, well-understood requirements (regulated, safety-critical). Agile is iterative — short sprints deliver working increments with continuous feedback and the freedom to change direction; a Scrum Master facilitates the process. Security must be planned into every sprint (security stories) or it gets skipped.

DevOps unifies development, QA and operations into one automated CI/CD flow for fast, frequent releases. Because speed is the point, security must be automated — adding it as a first-class, shifted-left member of the pipeline makes it DevSecOps / SecDevOps. Other terms to know: maturity models (SAMM, BSIMM, CMMI) measure how mature and repeatable your practices are, and canary releases limit the blast radius of a bad deployment.

Exam tip: “Requirements are fixed and fully documented” → Waterfall. “Frequent releases, automated pipeline” → DevOps. “Security automated into every build” → DevSecOps.

Sub-domain 8.2

Identify & apply security controls in the development ecosystem.

The tools, repositories and pipelines developers work in are themselves an attack surface — and the place to automate security testing.

8.2

Tooling, Repositories & Application Security Testing

SCMVersion control; integrity + audit trail
SASTStatic, white-box, on source — early
DASTDynamic, black-box, on running app
SCAScans dependencies for known CVEs

The development ecosystem is the programming languages, libraries, tool sets, IDEs, runtime and repositories developers use — each a control point. Software Configuration Management (SCM) — Git and friends — gives version control, code integrity, protected branches, signed commits, secret scanning and a full audit trail of who changed what. The CI/CD pipeline is where security testing is automated.

Know the testing types cold: SAST (static analysis) reads the source code — white-box, early, finds code-level flaws; DAST (dynamic analysis) attacks the running application — black-box, finds runtime and configuration flaws. IAST combines both from inside the app; SCA (software composition analysis) scans third-party dependencies for known vulnerabilities; and fuzzing throws malformed input to find crashes. Use several — each catches what the others miss.

Exam tip: SAST = Source (static, white-box); DAST = Dynamic/running (black-box). “Find flaws in a vendor's compiled binary you can't see inside” → DAST or fuzzing.

Sub-domain 8.3

Assess the effectiveness of software security.

Building controls is not enough — you must measure whether they actually work, and log the changes that could weaken them.

8.3

Auditing, Logging & Risk Analysis

Audit & logRecord changes; detect & attribute
Risk analysisFind, rank & mitigate code risk
VerificationDid we build it right? (spec)
ValidationDid we build the right thing? (need)

Effectiveness is proven, not assumed. Auditing and logging of changes gives the evidence trail — who changed which code or configuration, when, and why — so weakening changes are detected and attributable. Risk analysis and mitigation identifies, ranks and treats the risks in the codebase (code review findings, SAST/DAST results, dependency CVEs), feeding remediation back into the backlog.

Two words the exam pairs: verification asks “did we build the product right?” (against the specification — testing and review), while validation asks “did we build the right product?” (against the actual need). Both are needed; they are not the same.

Exam tip: Verification = against the spec (build it right); Validation = against the need (build the right thing). Do not swap them.

Sub-domain 8.4

Assess the security impact of acquired software.

Most software in an enterprise is bought or borrowed, not built — and you inherit every one of its vulnerabilities.

8.4

COTS, Open Source & Supply-Chain Risk

SourcesCOTS · open source · third-party · managed
InheritTheir vulnerabilities become yours
AssessVendor posture, CVEs, SBOM, licence
ContractSLAs, right to audit, breach notice

Acquired software comes as commercial off-the-shelf (COTS), open source, third-party components, and managed cloud services. Whichever it is, you adopt its flaws — the supply-chain risk. Manage it by assessing vendors (security posture, SOC 2 / ISO 27001, vulnerability history, patch cadence), tracking dependencies with a software bill of materials (SBOM), pinning known-good versions, and reviewing licences.

The contractual layer is the enforcement: contracts and SLAs define availability and support, plus security clauses — right to audit, breach-notification windows, data handling and liability. If a security obligation isn't written into the contract, it isn't guaranteed. Recent history (SolarWinds, Log4j) is why the exam takes this seriously.

Exam tip: Adopting a library or COTS product means adopting its vulnerabilities — the right answer usually involves vendor assessment, an SBOM/dependency review, and contractual SLAs, not just “trust the vendor”.

Sub-domain 8.5

Define & apply secure coding guidelines and standards.

The hands-on craft: defeat the classic weaknesses at the source-code level, validate all input, and secure APIs and databases.

8.5

The Classic Software Weaknesses

Buffer overflowInput exceeds buffer → overwrite memory
Injection (SQLi)Input executed as code → parameterise
XSS / CSRFClient script / forged request
TOCTOURace between check and use

Secure coding is about designing out the recurring flaws. A buffer overflow writes past a buffer into adjacent memory (often the return address) — countered by bounds checking, memory-safe languages, and runtime defences (ASLR, DEP, stack canaries). SQL injection and other injection flaws execute attacker input as code — defeated by parameterised queries and input validation. XSS runs script in other users' browsers (fix: output encoding); CSRF forges requests using a victim's live session (fix: anti-CSRF tokens, SameSite cookies). TOCTOU is a race condition between a permission check and use — found by code review, not scanning.

Round out the list: covert channels (storage/timing — minimise, can't eliminate), backdoors/trapdoors (remove maintenance hooks before release), memory/object reuse (zeroise buffers so residual data doesn't leak), citizen developers (govern low-code/no-code), and weak session management (random IDs, HttpOnly/SameSite cookies, timeouts, regenerate on login).

Exam tip: Most of these are input-handling or memory-safety flaws. “How are race conditions best found?” → code review (not scanning). “Stop SQL injection” → parameterised queries.
8.5

Input Validation, Obfuscation, APIs & Databases

Input validationServer-side, allow-list, never trust
ObfuscationSlows reverse eng — not primary
APIsREST (stateless) vs SOAP (WS-Security)
DatabasesACID + parameterised queries

Input validation is the foundation — never trust input; validate, sanitise and encode it server-side (client checks are for UX and trivially bypassed), preferring allow lists (known-good) over deny lists. Obfuscation (lexical, data, control-flow) makes code hard to reverse-engineer to protect IP — but it is a supplement, not a real control (security through obscurity is never primary).

APIs are a major modern attack surface: REST is lightweight and stateless (JSON over HTTPS, token auth), while SOAP is a heavier XML protocol with built-in WS-Security for message-level signing/encryption — secure both with authentication, input validation and rate limiting. For databases, know the DBMS components, the ACID properties that keep transactions reliable (Atomicity, Consistency, Isolation, Durability), concurrency control via locking, and that SQL injection is beaten by parameterised queries, least-privilege accounts, and polyinstantiation against inference.

Exam tip: Validate server-side with an allow list. REST = stateless JSON; SOAP = XML with native WS-Security. Obfuscation is never a primary control.
