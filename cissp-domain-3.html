Security Architecture and Engineering

Building security in by design — secure engineering principles, the formal security models, information-system capabilities, the vulnerabilities of modern architectures, cryptography and cryptanalysis, and the physical security of sites and facilities.

Sub-domains

Exam outline areas.

Domain 3 spans the engineering of security — ten outline areas from secure design principles and formal security models, through system capabilities and architecture vulnerabilities, to cryptography, cryptanalysis, and the physical security of sites and facilities.

3.1 — Secure design principles

Research, implement and manage engineering processes using secure design principles — least privilege, defense in depth, zero trust, secure defaults, privacy by design.

3.2 — Security models

The fundamental security models — Bell-LaPadula, Biba, Clark-Wilson, Brewer-Nash and the state machine / lattice foundations.

3.3 — Select controls

Select controls based upon systems security requirements — using evaluation criteria such as Common Criteria (EALs).

3.4 — Security capabilities of systems

Memory protection, TPM, secure boot, encryption/decryption, and trusted-computing capabilities.

3.5 — Assess & mitigate architecture vulnerabilities

Vulnerabilities of architectures and designs — client/server, databases, cloud, IoT, ICS/SCADA, virtualization, containers and serverless.

3.6 — Cryptographic solutions

Select and determine cryptographic solutions — symmetric, asymmetric, hashing, PKI, and key management.

3.7 — Cryptanalytic attacks

Methods of cryptanalytic attack — brute force, frequency analysis, known/chosen-plaintext, side-channel, and more.

3.8 — Site & facility design principles

Apply security principles to site and facility design — including CPTED and layered physical defense.

3.9 — Site & facility security controls

Design the controls — perimeter, power, HVAC, fire detection & suppression, and environmental protection.

3.10 — Information system lifecycle

Manage the information system lifecycle — from requirements and development through operation to secure disposal.

Sub-domain 3.1

Research, implement & manage engineering processes using secure design principles.

Good architecture starts with a handful of timeless principles. Every later control is easier — and cheaper — when these are baked in at design time.

3.1

Economy of Mechanism, KISS & Open vs Closed Systems

Economy of mechanismKeep it small & simple — complexity breeds flaws
Unused featuresRemove in DESIGN — not just disable
Open systemStandards-based; security from implementation
ObscurityNEVER a valid primary strategy

Economy of mechanism (one of Saltzer & Schroeder's classic eight principles) says security mechanisms should be as simple and small as possible: every extra feature is extra attack surface and something else to audit, test and misconfigure. Its corollary — avoid unnecessary functionality — means unneeded features are removed at design time, not merely shipped disabled (a disabled feature is one config change from becoming a vulnerability). Keep It Simple (KISS) extends this to structure: separate the security-critical functions into small, isolated, rigorously tested components.

Open systems are built on agreed industry standards — more widely understood and more widely targeted, so their security must come from robust implementation, secure coding and defense in depth. Closed systems are proprietary and rely partly on security through obscurity — the exam's point is unambiguous: obscurity can supplement defences but is never a valid primary strategy, because the moment the secret leaks, security collapses. Contrast that with Kerckhoffs' principle in cryptography: assume everything except the key is public.

Dig deeper: Saltzer & Schroeder — The Protection of Information · NIST SP 800-160 (Systems Security Engineering)

Exam tip: A vendor claiming their product is secure “because nobody knows how it works” is the classic obscurity trap — the answer is always that obscurity alone is never sufficient.
3.1

The Core Principles — Least Privilege to Shared Responsibility

Least privilegeMinimum rights, minimum time
Defense in depthLayered controls; no single point of failure
Fail securelyDefault deny on failure (unless life safety)
SoDNo single person completes a critical task

Threat modeling asks “what can go wrong?” at design time — using STRIDE to classify threats, PASTA to simulate attacks against business impact, or DREAD to rank them (all covered in Domain 1; here they drive architecture decisions). Least privilege grants only the rights the job needs, for only as long as needed — ideally Just-In-Time. Defense in depth layers administrative, technical and physical controls so one failure never reaches the asset. Secure defaults means the system ships in its most secure configuration: default-deny, features off, no default passwords.

Fail securely is the failure-mode principle: when a control fails, it should fail CLOSED (blocking access) — the standard for secure design — unless human life requires fail open (emergency exits). Separation of duties splits critical tasks so fraud requires collusion; privacy by design embeds privacy as the DEFAULT setting from the start (now codified in GDPR Article 25); and shared responsibility recognises that in cloud and outsourced models, security duties are split by contract — but accountability never transfers.

Dig deeper: OWASP Secure Design Principles · GDPR Art. 25 — Data protection by design & default

Exam tip: A door controller loses power and there is no life-safety constraint → fail secure (locked). Any question where people could be trapped → fail safe/open. People > property, every single time.
3.1

Zero Trust — Never Trust, Always Verify

The tupleIdentity + device posture + context
Per-requestEvery request re-verified; no carried trust
Three pillarsAlways Verify · Least Privilege · Assume Breach
StandardNIST SP 800-207

Zero Trust rejects the idea of a “trusted internal network”. Nothing is automatically trusted for being inside the perimeter: every request is authenticated, authorized and encrypted, evaluated against the decision tuple of user identity + device posture + context (time, location, prior behaviour). A successful login this morning means nothing this afternoon — past trust is never carried forward.

The three pillars: Always Verify (per-request checks), Least Privilege (minimum access, ideally Just-In-Time), and Assume Breach (design as if the attacker is already inside — segment aggressively to limit the blast radius). The classic failure Zero Trust fixes: once on the VPN, a user could reach everything. That flat internal trust violates both Assume Breach and Least Privilege. Contrast with the legacy trust-but-verify stance — allow, then audit the logs afterwards.

Dig deeper: NIST SP 800-207 — Zero Trust Architecture · CISA Zero Trust Maturity Model

Exam tip: Scenario: a familiar user, but an unrecognised device in a new country. Zero Trust re-evaluates the whole tuple on that request — identity alone never wins access.
3.1

Confinement, Bounds & Isolation

BoundsThe LIMITS set on a process (memory, resources)
ConfinementThe RESTRICTION of what it may touch
IsolationThe ENFORCEMENT of that restriction
PayoffA compromised process can't reach its neighbours

Three linked containment concepts the exam distinguishes precisely. Bounds set strict limits on the resources a process may use — properly enforced memory bounds are what prevent a buffer overflow from writing past its allocation. Confinement is the act of restricting a process to those bounds, limiting what it can read, write or invoke. Isolation is the physical or logical means by which confinement is enforced, so a compromised process cannot read, corrupt or interfere with another's data.

Remember the chain: bounds set the limits → confinement is the restriction → isolation enforces it. Sandboxes, containers and VMs are all isolation mechanisms at different strengths — and Domain 3's memory protections (process isolation, hardware segmentation, ASLR) are this principle implemented in silicon and OS code.

Exam tip: “A program wrote past the end of its buffer” → the concept not enforced is bounds. If the question asks what stops process A reading process B's memory → isolation.

Sub-domain 3.2

Understand the fundamental concepts of security models.

A security model translates a policy (“no leaking secrets”) into precise, enforceable rules (“no write down”). Know each model's GOAL first — the rules follow.

3.2

Model Families — State Machine, Information Flow, Noninterference, Lattice

State machineSecure in every state & transition
Information flowControls WHERE data may move
NoninterferenceHigh actions invisible to low subjects
LatticeLevels + labels; bounds of access

Before the named models, know the four foundations. A state machine model proves a system is secure by showing every possible state and every transition between states preserves the security policy — if it starts secure and every transition is secure, it stays secure. Information flow models (the basis of both Bell-LaPadula and Biba) control where information is allowed to move, blocking flows that violate policy.

Noninterference demands that actions at a higher level are completely invisible at lower levels — a low-clearance user should not even be able to infer that a high-level action occurred (this is the theory behind defeating covert channels). Lattice-based models arrange classification levels and compartments into a lattice; every subject/object pair has a least upper bound and greatest lower bound defining what access is possible. BLP and Biba are both lattice, information-flow, state-machine models — which is why those descriptors appear together in exam answers.

Exam tip: “A user deduces classified activity from system slow-downs” → a noninterference failure (and a timing covert channel). Match the family before the named model.
3.2

Bell-LaPadula vs Biba — Mirror Images

BLP goalCONFIDENTIALITY (military heritage)
BLP rulesNo Read Up · No Write Down
Biba goalINTEGRITY (mirror of BLP)
Biba rulesNo Read Down · No Write Up

Bell-LaPadula protects secrets. The Simple Security Property = No Read Up (a SECRET subject cannot read TOP SECRET); the Star (*) Property = No Write Down (a SECRET subject cannot write into an UNCLASSIFIED file — that would leak); the Strong Star variant confines subjects to read/write at exactly their own level. The first mathematically proven model, built for the US military.

Biba inverts it for integrity: Simple Integrity = No Read Down (don't base decisions on dirty, low-integrity data) and Star Integrity = No Write Up (a low-integrity source must not corrupt high-integrity records), plus the Invocation Property — a subject may not invoke (request services of) a higher-integrity subject. Know Biba's weaknesses cold: it addresses integrity ONLY (no confidentiality or availability), doesn't manage access control administration, doesn't prevent covert channels, and has no mechanism for changing classification levels.

Exam tip: Memory hook — in BLP the "S" rules protect Secrecy on reads; in Biba the rules flip direction. If the scenario worries about leaks → BLP; about corruption/trustworthiness → Biba.
3.2

Clark-Wilson, Brewer-Nash & the Access-Matrix Models

Clark-WilsonSubject → Program → Object; SoD
Brewer-NashChinese Wall; blocks conflicts of interest
Graham-Denning8 rules for subjects/objects/rights
HRUCan rights LEAK? (the safety problem)

Clark-Wilson is commercial integrity: subjects never touch data directly — every change goes through a certified Transformation Procedure (the access triple Subject → Program → Object), well-formed transactions keep data consistent, an Integrity Verification Procedure audits it, and enforced separation of duties means fraud requires collusion. Think banking: nobody edits the ledger directly, and no one both creates and approves a payment.

Brewer-Nash (Chinese Wall) is the information-flow model against conflicts of interest: the moment you access Client A's data, a wall dynamically drops between you and every competitor of Client A in the same conflict class — access rights change based on your own history. Graham-Denning defines the eight protection rules for securely creating/ deleting subjects and objects and granting/transferring/revoking rights; Harrison-Ruzzo- Ullman (HRU) extends it to ask whether a right can ever leak to an unauthorised subject — the “safety problem”, undecidable in the general case. Lipner simply combines BLP + Biba when you need confidentiality AND integrity together.

Exam tip: “A consultant who touched Client A is suddenly blocked from Client A's competitor” → Brewer-Nash, every time. “Users must go through validated programs, never direct to data” → Clark-Wilson.

Sub-domain 3.3

Select controls based upon systems security requirements.

How do you know a product is as secure as the vendor claims? Independent evaluation against a published standard — and a formal management decision to accept the residual risk.

3.3

Common Criteria — PP, ST, TOE & the EAL Ladder

PPProtection Profile — what a CATEGORY requires
STSecurity Target — what the VENDOR claims
TOETarget of Evaluation — the product tested
EAL4Commercial max; EAL7 = formally verified

Evaluation criteria evolved in three generations: TCSEC (the US Orange Book — confidentiality only, ratings D through A1), ITSEC (Europe — added integrity and availability, split Functionality from Effectiveness/assurance), and today's global standard Common Criteria (ISO/IEC 15408). In CC vocabulary: the Protection Profile states what a product category must do (“what is required”), the Security Target is the vendor's specific claim (“what is promised”), and the Target of Evaluation is the actual product tested (“what is evaluated”).

The EAL ladder: EAL1 functionally tested → EAL2 structurally tested → EAL3 methodically tested & checked → EAL4 methodically designed, tested and reviewed (the commercial ceiling) → EAL5 semiformally designed → EAL6 semiformally verified → EAL7 formally verified (mathematical proof). Critical nuance: a higher EAL means more rigorous evaluation of the claims — not necessarily more security features.

Dig deeper: Common Criteria Portal · ISO/IEC 15408

Exam tip: PP = requirements, ST = claims, TOE = product. And EAL4 is “the most common for commercial products” — a recurring right answer.
3.3

Certification, Accreditation & Authorization to Operate

CertificationTECHNICAL evaluation of controls
AccreditationMANAGEMENT acceptance of risk
ATOSenior official's risk-based approval
OrderCertify first, then accredit

Certification is the comprehensive technical evaluation of a system's security controls against requirements — done by assessors, producing evidence. Accreditation is the formal management decision to accept the residual risk and authorise operation, made on the strength of that evidence. Certification evaluates; accreditation approves — and it must be repeated when significant changes occur.

In US federal practice (NIST RMF, SP 800-37) the accreditation decision is the Authorization to Operate (ATO), granted by the Authorizing Official. The four possible decisions: Authorization to Operate, Common Control Authorization, Authorization to Use (accepting another agency's authorization), and Denial of Authorization.

Dig deeper: NIST SP 800-37 Rev 2 — Risk Management Framework

Exam tip: Who accepts the risk? Always management (the AO) — never the assessor, never the engineer. Technical people certify; leaders accredit.

3.4

CPU & Memory Fundamentals — Rings, States & Protection

Ring 0 / Ring 3Kernel (highest priv) / apps (lowest)
MultitaskingOne CPU switching fast (illusion)
MultiprocessingMultiple CPUs — TRUE parallelism
ASLRRandomises memory layout each boot

CPU anatomy: the ALU performs the actual mathematical/logical operations; the Control Unit directs instructions but executes nothing itself; registers are the CPU's tiny high-speed internal memory. Execution vocabulary: multitasking = one CPU context-switching fast enough to fake parallelism; multiprocessing = true parallel execution on multiple CPUs/cores; multithreading = lightweight threads sharing one process's memory. The protection ring model puts the kernel in Ring 0 (full hardware access) and user applications in Ring 3 — higher number, lower privilege — matching the processor's supervisor vs problem states and the OS's kernel vs user modes.

Memory protection is layered: process isolation gives each process its own address space; hardware segmentation enforces boundaries in silicon (stronger than OS logic alone); and ASLR randomises where the stack, heap and libraries load at each boot, so exploits can't rely on predictable addresses. Also know virtual memory (paging/swap — secrets can persist in the swap file) and data hiding (each layer exposes only what the layer above needs).

Exam tip: “Single-core CPU appears to run many apps at once” → multitasking, not multiprocessing. “Which component adds two numbers?” → the ALU. “Defeats predictable-address exploits” → ASLR.
3.4

TCB, Reference Monitor & Security Kernel

TCBEverything you TRUST — keep it small
Reference monitorThe concept: mediate EVERY access
Security kernelThe implementation of the monitor
Three rulesAlways enforce · non-bypassable · verifiable

The Trusted Computing Base is ALL the hardware, software and firmware critical to security — bounded by the security perimeter. If any TCB component is compromised, the whole system is at risk, which is why the cardinal rule is keep the TCB small: every component a vendor adds to the trusted core enlarges the attack surface.

The Reference Monitor is the abstract “security brain”: every access by every subject to every object must be checked, the checks can never be bypassed, and the mechanism must be small enough to verify. The Security Kernel is its concrete implementation — the core OS code inside the TCB that actually makes and enforces the access decisions. One-line hierarchy: TCB = what you trust; Reference Monitor = what must be enforced; Security Kernel = how it is enforced.

Exam tip: “Always enforced, cannot be bypassed, verifiable” — those three properties in a question stem = Reference Monitor, not the kernel (the kernel is the implementation).
3.4

Hardware Roots of Trust — TPM vs HSM

TPMChip ON the motherboard; per-device
HSMNetwork appliance; high-volume keys
TPM functionsAttestation · Binding · Sealing
StandardFIPS 140-3 validates crypto modules

A TPM (Trusted Platform Module) is a cryptographic coprocessor chip integrated on the motherboard — the hardware root of trust for secure/measured boot and local data protection. Its three exam functions: attestation (proving system integrity to a third party via cryptographic hashes of the boot chain), binding (tying encrypted data to this specific device), and sealing (releasing the key only when the hardware is in a known-good, untampered state — how BitLocker works).

An HSM (Hardware Security Module) is a dedicated, tamper-responsive network appliance for high-volume cryptographic operations and centralized key management — keys are generated, used and destroyed inside its boundary and can be non-exportable. CA root keys and payment-processing keys live in HSMs, validated against FIPS 140-3. The distinction is pure exam gold: TPM = integrated chip, one device; HSM = standalone appliance, enterprise scale.

Dig deeper: TCG TPM 2.0 Specification · NIST FIPS 140-3

Exam tip: “Key released only if the machine boots into a known-good state” → sealing. “Millions of key operations, centrally managed” → HSM. Don't confuse attestation (prove to others) with sealing (condition the key).

Sub-domain 3.5

Assess & mitigate the vulnerabilities of security architectures, designs & solution elements.

Every architecture — from a single binary to a global cloud — has characteristic weaknesses. This is the exam's biggest 3.x area: client to server, database to SCADA, VM to serverless.

3.5

Code-Level Flaws — Race Conditions, TOCTOU, Buffer Overflows, Covert Channels

TOCTOUSwap between the CHECK and the USE
Found byCODE REVIEW — not scanning
Buffer overflowInput exceeds allocation; overwrites memory
Covert channelsStorage vs Timing; minimise, can't eliminate

A race condition exists when the outcome wrongly depends on the timing of concurrent access to a shared resource; TOCTOU (Time-of-Check to Time-of-Use) is its most common form — the attacker swaps the resource (say, replaces a checked file with a symlink to /etc/shadow) in the microseconds between authorization and use. Because these are logic flaws, they are found by code review, not vulnerability scanning — a detail the exam tests directly. Mitigate by shrinking the window and re-validating at the moment of use.

A buffer overflow happens when software accepts more input than the buffer holds and the excess overwrites adjacent memory — countered by good coding practice, source-code scanners, developer education and strongly-typed languages, plus runtime defences (bounds enforcement, ASLR). Covert channels move information in ways policy never intended: storage channels signal by modifying stored objects, timing channels by modulating event timing. In complex systems they can only be minimised through covert-channel analysis at design time — never fully eliminated. Related leakage: emanations (electromagnetic leakage — see TEMPEST in 3.7/3.9).

Dig deeper: CWE-367 (TOCTOU) · OWASP — Buffer Overflow

Exam tip: “How are race conditions best identified?” → code review. Not pen testing, not vulnerability scanning. It appears verbatim.
3.5

Databases & Data Inference — Aggregation, Inference, Polyinstantiation

AggregationCombine harmless pieces → sensitive whole
InferenceDEDUCE secrets from visible data
PolyinstantiationDifferent versions per clearance level
Also knowViews, content-dependent access control

Databases fail in ways firewalls can't see. Aggregation: many individually harmless records, combined, become sensitive (a full troop-movement picture from unclassified logistics rows). Inference: a user deduces information beyond their clearance from what they can legitimately see (the department's budget jumped the month a classified project started). The classic defence is polyinstantiation — keeping multiple versions of the same record at different classification levels so lower-cleared users see a plausible cover story instead of a telling gap.

Round out with standard database controls: views restrict which columns/rows a user sees, content- and context-dependent access control filter by data value and by access history, and noninterference (3.2) is the formal theory underneath all of it.

Exam tip: Aggregation = combining; inference = deducing. If the question says the user “put pieces together” it's aggregation; if they “figured out” something hidden, it's inference — and polyinstantiation is the named fix.
3.5

ICS / SCADA, Embedded & IoT

DCSProcess control · state-driven · one site
SCADAData gathering · event-driven · wide area
Best controlAIR-GAP when patching isn't feasible
ZigbeeIoT mesh; close range; 128-bit symmetric

Industrial Control Systems run physical processes, so their priority order flips: availability and integrity before confidentiality — and attacks have physical-world consequences (valves, turbines, power grids). Distinguish DCS (Distributed Control System: state-driven process control across a limited area, like one plant) from SCADA (Supervisory Control and Data Acquisition: event-driven data gathering and monitoring across large geographic areas, like a pipeline). Patching is often impossible — systems run non-stop and field devices may be physically unreachable — so air-gapping (with network segmentation and one-way data diodes) is the primary compensating control.

Embedded systems are dedicated-function, limited-power platforms with very long service lives — the same constraints in miniature. For IoT, the recurring issues are weak authentication, unpatchable firmware and default credentials; keep IoT on isolated network segments. Know Zigbee by name: a low-power, low-throughput, close-proximity IoT protocol whose communications are encrypted with a 128-bit symmetric algorithm.

Dig deeper: NIST SP 800-82 Rev 3 — OT Security · CISA — ICS Security

Exam tip: DCS = process, state-driven, small footprint; SCADA = data, event-driven, huge footprint. And when the stem says “can't patch the PLCs”, the answer is air-gap/segment — not “patch harder”.
3.5

Virtualization, Containers, Serverless & Microservices

Type 1 / Type 2Bare-metal (servers) / hosted (desktops)
VM sprawlUnmanaged VM growth = unpatched targets
ContainersShare the host kernel — secure the HOST
ServerlessCSP runs ALL infra; event-driven, stateless

A Type 1 (bare-metal) hypervisor runs directly on hardware — small attack surface, the data-centre standard; Type 2 (hosted) runs atop a host OS and inherits all its flaws — fine for desktops and labs. The hypervisor is absolute: compromise it (VM escape) and every guest falls. The governance failure is VM sprawl — VMs multiplying beyond the team's ability to track and patch. Containers package an app with its dependencies but share the host OS kernel, so the single most critical control is hardening the host (plus using only trusted, scanned images — compromised images/repositories are the primary container risk).

Serverless: you write code, the CSP manages ALL infrastructure — event-driven execution in stateless, short-lived containers. Your attack surface shrinks (the CSP patches), but you lose visibility, and over-permissioned function roles and event injection become the new risks. Microservices build the app as small, loosely-coupled independent services — which multiplies inter-service network connections, each of which must be authenticated and encrypted, with security designed in from day one. And the exam's favourite distinction: microservices ≠ containers — a microservice may run without a container, and a container can hold a monolith.

Dig deeper: NIST SP 800-190 — Container Security · OWASP Serverless Top 10

Exam tip: “All containers on one host — what do you secure first?” → the host OS kernel. “Dozens of untracked, unpatched VMs” → VM sprawl.
3.5

Cloud — Models, Multitenancy, CASB & Mobile Fleets

Greatest concernMULTITENANCY (shared hardware)
Data residencyLEGAL review first — encryption ≠ compliance
CASBPolicy enforcement between user & cloud
MobileCOMS best → COPE → CYOD → BYOD least

The service models split responsibility (IaaS: you manage OS-up; PaaS: apps-up; SaaS: users and data only), the deployment models split tenancy (public, private, community, hybrid) — and SaaS-on-public-cloud gives the most cost saving with the least administrative overhead. The customer always remains accountable for data, identity and configuration. The single greatest cloud security concern the exam names is multitenancy — your data on the same physical hardware as strangers', separated only logically. And data residency: moving data across borders triggers the destination's jurisdiction — encryption does NOT exempt you from sovereignty laws; a legal review comes first.

A CASB (Cloud Access Security Broker) sits in the traffic path between users and cloud services: real-time monitoring, policy enforcement before data reaches the cloud, scanning of data at rest, and exposure of shadow IT. Endpoint fleet policy, strongest to weakest: COMS (corporate-owned, fully controlled — best posture) → COPE (corporate-owned, personally enabled) → CYOD (choose from an approved list) → BYOD (least control — compensate with MDM + MFA, remote wipe and containerisation).

Dig deeper: NIST SP 800-145 — Cloud Definition · CSA Security Guidance

Exam tip: “Biggest worry about public cloud?” → multitenancy. “Tool that finds shadow IT and enforces policy before upload?” → CASB. “Is encryption enough for residency laws?” → No — legal review.

Sub-domain 3.6

Select & determine cryptographic solutions.

The heaviest-weighted slice of Domain 3. Master the PAIN framework, the symmetric/asymmetric split, hashing, PKI and the key lifecycle — and always know WHICH key does WHAT.

3.6

Goals & Vocabulary — PAIN, Kerckhoffs, Confusion & Diffusion

P·A·I·NPrivacy · Authentication · Integrity · Non-repudiation
PrivacyEncrypt with RECEIVER's public key
AuthenticationSign with SENDER's private key
IntegrityHASHING — never encryption

The PAIN memory aid maps each cryptographic goal to its mechanism and its key: Privacy (confidentiality) = encrypt with the receiver's public key; Authentication = sign with the sender's private key; Integrity = hashing (never encryption — the single most common trap in this domain); Non-repudiation = a hash encrypted with the sender's private key, i.e. a digital signature. Note that only asymmetric mechanisms can give non-repudiation — a shared symmetric key can never prove which holder acted.

Core vocabulary: Kerckhoffs' principle — the system must stay secure with everything public except the key. Work factor — the cost/time to break the crypto; it must exceed the data's useful life, and each added key bit doubles it. IV/nonce — a random/unique start value so identical plaintexts encrypt differently; nonce reuse is catastrophic. Shannon's pair: confusion (substitution hides the key-ciphertext relationship) and diffusion (permutation spreads each plaintext bit's influence); their visible result is the avalanche effect — flip one input bit and about half the output changes. Key clustering (two keys producing the same ciphertext) is an algorithm weakness.

Dig deeper: NIST Cryptographic Standards

Exam tip: Read every crypto stem asking “which key?” against PAIN. Encrypt-to-someone = their public key. Prove-it-was-you = your private key. “Verify integrity only” = hash, no keys at all.
3.6

Symmetric Cryptography — Algorithms, Modes & the Key-Count Problem

Key countn(n−1)/2 — 10 users = 45 keys
AES128-bit block; 128/192/256 keys — the standard
DES / 3DESBroken (56-bit) / retired stopgap
ECBPatterns survive — never use

Symmetric crypto uses ONE shared secret key — fast and ideal for bulk data, but crippled by distribution: every pair needs its own key, so n(n−1)/2 keys (10 people = 45) and the key must travel out-of-band. Stream ciphers XOR a pseudorandom keystream bit-by-bit (RC4 — now prohibited); block ciphers process fixed blocks (most algorithms). The roster: DES broken (56-bit key, brute-forced since 1998); 3DES the transitional stopgap (retired); AES/Rijndael the current standard (128-bit blocks; 128/192/256-bit keys); plus recognition names Blowfish/Twofish, CAST-128, SAFER (Bluetooth), RC5/RC6.

Modes decide how blocks chain: ECB encrypts blocks independently so patterns survive (the famous penguin) — never use it; CBC chains with the previous block + IV; CFB/OFB run the block cipher as a stream; CTR turns a counter into keystream — parallel, modern, and with authentication becomes GCM (authenticated encryption). Mode choice, not algorithm choice, is what usually sinks implementations.

Dig deeper: FIPS 197 — AES · NIST SP 800-38A — Block Cipher Modes

Exam tip: The key-count arithmetic appears constantly: symmetric = n(n−1)/2, asymmetric = 2n. If 10 users need 45 keys, adding the 11th adds 10 more.
3.6

Asymmetric Cryptography — RSA, ECC, Diffie-Hellman, ElGamal & Quantum

RSAFactoring two large primes; 2048-bit floor
ECCHighest strength/bit — 256 ≈ RSA-3072
Diffie-HellmanKey EXCHANGE only; MITM if unauthenticated
QKDQuantum key exchange; provably eavesdrop-proof

Asymmetric crypto uses a mathematically linked pair — public key shared freely, private key never. RSA rests on the difficulty of factoring the product of two large primes (1024/2048/4096-bit keys; slow, so it wraps session keys rather than bulk data). ECC delivers the highest strength per bit — 256-bit ECC ≈ 3072-bit RSA — making it the choice for mobile, smart cards and IoT. Diffie-Hellman is key exchange only: two parties derive a shared secret over an open channel, but with no authentication it falls to man-in-the-middle — pair it with certificates. ElGamal extends DH's discrete-log maths to encryption and signatures (cost: ciphertext doubles in size). In practice everything is hybrid: asymmetric handshake, symmetric session (TLS).

Forward-looking: a large quantum computer running Shor's algorithm breaks RSA/DH/ECC, which is why NIST has standardised post-quantum algorithms (ML-KEM/ML-DSA, FIPS 203-205). QKD (Quantum Key Distribution) uses photon physics to exchange keys with provable eavesdropping detection — but remember: it distributes keys only; it does not encrypt messages.

Dig deeper: NIST Post-Quantum Cryptography

Exam tip: “Small keys, constrained device” → ECC. “Agree a key over an open network” → DH — with the MITM caveat. “Provably secure key sharing via photons” → QKD, key exchange only.
3.6

Hashing, Digital Signatures & Blockchain

HashOne-way, fixed-length, NO keys
SignatureHash encrypted with sender's PRIVATE key
Verified withSender's PUBLIC key
SaltingUnique random value; defeats rainbow tables

A hash is a one-way function: variable input → fixed digest, using no keys, serving integrity only. Status board: MD5 and SHA-1 broken (collisions); SHA-2 the workhorse; SHA-3 the different-maths hedge. The birthday attack finds two messages with the same digest (collisions come at 2^(n/2), not 2^n) — defend with stronger algorithms and salting: a unique random value per user, added before hashing and changed on password update. A keyed hash (HMAC) adds authenticity but still not non-repudiation.

A digital signature = the message hash encrypted with the sender's private key; the recipient verifies by decrypting with the sender's public key and comparing digests — delivering authenticity, integrity AND non-repudiation in one mechanism (DSA/ECDSA sign only, never encrypt). Blockchain applies these primitives — chained hashes across a distributed, decentralised ledger — to give integrity, non-repudiation and tamper-resistance in distributed environments.

Dig deeper: FIPS 180-4 — Secure Hash Standard · FIPS 186-5 — Digital Signature Standard

Exam tip: Integrity comes from HASHING, never from encryption — ISC²'s favourite trap. And the signature key directions: sign = sender's private; verify = sender's public. Drill until reflexive.
3.6

PKI & Key Management — CA/RA, CRL/OCSP, Escrow & Split Knowledge

CA vs RACA SIGNS; RA only verifies identity
CRL vs OCSPPeriodic list (delay) vs real-time query
StaplingServer pre-fetches OCSP proof; cuts CA load
Split knowledgeNo one holds the whole key

PKI creates, manages, distributes and revokes X.509 certificates — binding identities to public keys so strangers can trust each other. Roles: the CA is the trusted root that creates and signs certificates; the RA establishes and verifies the requester's identity but never signs; intermediates do the daily issuing so the offline root stays safe; and for true non-repudiation PKI needs a reliable time source for secure timestamping. Revocation: a CRL is a periodically published blacklist — inherently stale between updates; OCSP queries status in real time; OCSP stapling has the web server pre-fetch the signed response and attach it to the handshake, cutting both latency and CA load. Pinning hard-codes the expected certificate/key in the client, defeating rogue-CA MITM.

Key management lifecycle: generation (CSPRNG/HSM entropy) → distribution (DH, out-of-band, or hybrid wrap) → storage (TPM/HSM) → rotation (limit each key's cryptoperiod) → disposition (crypto-shredding, verified destruction) → recovery. The recovery controls: split knowledge (two+ people each hold a portion; none can reconstruct alone), dual control (two people must act together — divides the ACTION, where split knowledge divides the SECRET), and key escrow (a trusted third party holds copies for emergency/legal recovery).

Dig deeper: NIST SP 800-57 — Key Management · RFC 5280 — X.509 PKI

Exam tip: “Who verifies identity vs who signs?” → RA verifies, CA signs. “Real-time revocation with less CA load?” → OCSP + stapling. Split knowledge vs dual control = secret vs action.

Sub-domain 3.7

Understand methods of cryptanalytic attacks.

Crypto rarely falls to mathematics — it falls to implementations, physics and people. Rank the classic attacks by attacker knowledge, then split physical attacks into passive vs active.

3.7

The Knowledge Ladder & Implementation Attacks

HardestCiphertext-only (least info)
LadderCO → Known-PT → Chosen-PT → Chosen-CT
Brute forceDefence = increase KEY LENGTH
Pass-the-HashAuthenticate with the stolen hash itself

Rank attacks by what the attacker HAS: ciphertext-only (hardest — only intercepts); known-plaintext (plaintext-ciphertext pairs — Bletchley's cribs); chosen-plaintext (attacker gets chosen inputs encrypted); chosen-ciphertext (gets decryptions too — the strongest position, needs access to the decryption system). Linear & differential cryptanalysis are the academic block-cipher attacks; factoring targets RSA's modulus directly. Against brute force the single most effective defence is simply increasing key length — each bit doubles the work.

Implementation-level: Pass-the-Hash captures an NTLM hash (Mimikatz) and authenticates with the hash directly — no cracking needed, which is why hashes deserve password-grade protection. Replay re-sends captured packets — killed by timestamps, nonces and sequence numbers. Temporary files, swap and crash dumps leak plaintext long after the “encrypted” work ended. Ransomware weaponises encryption against you — primarily an attack on data integrity/availability. Password-focused: dictionary attacks try likely words; rainbow tables use precomputed hash chains — completely neutralised by salting.

Dig deeper: MITRE ATT&CK — Pass the Hash

Exam tip: “Which attack is hardest for the attacker?” → ciphertext-only. “Best brute-force defence?” → longer keys. “Authenticated with a hash, never cracked it” → Pass-the-Hash.
3.7

Physical Attacks — Side-Channel vs Fault Injection vs TEMPEST

Side-channelPASSIVE — observe timing/power/EM
Fault injectionACTIVE — voltage spikes, clock glitches
TEMPESTIntercept EM emanations at a distance
Faraday cageFully absorbs/contains the emissions

The examiner's favourite split: if the attacker only watches, it's passive; if they change the device's behaviour, it's active. Side-channel attacks are passive — observing timing variations, power consumption (SPA/DPA), cache access or EM emissions to recover keys without ever touching the algorithm's mathematics. They're hard to detect because nothing is disturbed. Fault injection is active — deliberately injecting voltage spikes or clock glitches to force exploitable errors; countered by shielding, output blocking, and erasing keys after N failed attempts.

TEMPEST / emanation eavesdropping intercepts the electromagnetic emissions leaking from equipment (the van-Eck “read the screen from the parking lot” attack). Defences in strength order: a Faraday cage (metal-mesh enclosure that fully absorbs EM signals), white noise (broadcast false signals to mask the real ones), and control zones (restrict the physical areas signals can reach). Social routes complete the picture: purchase-key (bribe/buy the secret) and rubber-hose (coercion) — countered procedurally by split knowledge and dual control, because no algorithm resists a wrench.

Exam tip: Measuring how long operations take = side-channel (passive). Glitching the voltage = fault injection (active). Van in the parking lot reading EM = TEMPEST; the full block is a Faraday cage.

Sub-domain 3.8

Apply security principles to site & facility design.

Physical security begins on the architect's drawing board — shaping behaviour with CPTED, hardening what remains, and sequencing controls in the order an intruder meets them.

3.8

CPTED, Target Hardening & the Six D's

CPTEDPSYCHOLOGICAL — shapes behaviour
3 strategiesAccess Control · Surveillance · Territorial
Target hardeningPHYSICAL denial — locks, bollards, bars
Six D'sDeter → Deny → Detect → Delay → Determine → Decide

CPTED (Crime Prevention Through Environmental Design) reduces crime by shaping how people feel and behave in a space — legitimate users feel ownership, intruders feel exposed. Its three strategies: Natural Access Control (physical elements guide and restrict entry/exit paths — hedges funnelling visitors past reception); Natural Surveillance (maximise visibility — low landscaping, open sightlines, windows on walkways); Territorial Reinforcement (clear boundaries create ownership — signage, pavement changes, defined property lines). Target hardening is the contrast: physical barriers that simply DENY access — bollards, reinforced doors, grilles. A strong programme layers both: CPTED discourages and channels; hardening denies what remains.

The facility planning sequence: risk assessment → critical path analysis → site selection (cost, location, size, neighbours) → facility design (CPTED) → control implementation → KPI monitoring. And the physical-security order of operations — the Six D's: Deter (discourage the attempt), Deny (block it), Detect (sense it), Delay (slow it), Determine (assess what it is), Decide (choose the response). Strict order — the exam tests the sequence.

Dig deeper: International CPTED Association

Exam tip: CPTED = behavioural/psychological; target hardening = physical/absolute. Open sightlines → Natural Surveillance; funnelled paths → Natural Access Control; boundary markers → Territorial Reinforcement.
3.8

Server Rooms, SCIFs & Restricted Areas

LocationCORE of the building — no windows
EntryONE exclusive controlled entrance
AirPositive pressurisation — air flows OUT
SCIFClearance-only compartmented facility

The server room is a single point of catastrophic failure: physical access to the console bypasses every logical control (pull drives, reset credentials, plant implants). Design rules the exam expects verbatim: locate it in the core of the building (never an exterior wall), give it one exclusive controlled entry (single entrance = best control; multiple entrances exist only for emergency evacuation), no windows, and positive pressurisation — internal air pressure higher than outside, so air flows outward when doors open and dust, smoke and contaminants can never drift in.

A SCIF (Sensitive Compartmented Information Facility) goes further: a hardened, access-controlled area for top-secret work requiring clearance and per-visit approval — typically with TEMPEST shielding, no personal electronics, and continuous monitoring. Same philosophy, government-grade execution.

Exam tip: An architect proposing a server room on an exterior wall, with windows and two entrances, is a walking exam question — every one of those choices is wrong. Core, single entry, no windows, positive pressure.

Sub-domain 3.9

Design site & facility security controls.

The concrete controls: perimeter and entry, power and climate, fire detection and suppression — with life safety governing every choice.

3.9

Perimeter & Entry — Fences, Alarms, Locks, Mantraps

Fences3-4ft casual · 6-7ft most · 8ft+wire determined
AlarmsDeterrent · Repellent · Notification (silent)
Fail statesFail-secure = server rooms; fail-safe = people
MantrapStrongest vs tailgating & piggybacking

Fence heights are pure memorisation: 3–4 ft deters casual trespassers only; 6–7 ft deters most intruders; 8+ ft topped with barbed/razor wire is required for determined intruders. Intrusion alarms come in three roles: deterrent (make the attack harder/discourage it), repellent (sirens and lights to drive the intruder off), and notification (a SILENT alert so responders act without tipping the intruder). Locks are delay controls, not absolute barriers — every lock (combination, cipher, smart) can be picked or shimmed; they buy time for detection and response.

Electronic locks need a declared fail state: fail-secure (stays locked on power loss — required for server rooms) vs fail-safe (opens — wherever human egress matters). Cards: magnetic-stripe is trivially cloned and insufficient; smart cards carry a chip and support MFA. Entry abuse: tailgating = following someone through WITHOUT their knowledge; piggybacking = WITH their consent. A turnstile merely manages throughput; a mantrap — interlocked double doors isolating one person per cycle, often guard-monitored — is the strongest control against both.

Exam tip: Tailgating vs piggybacking is decided by one word: consent. And “strongest control against both” is always the mantrap — never the turnstile.
3.9

Power & Environment — Anomalies, HVAC, Humidity

High voltageSpike (momentary) · Surge (prolonged)
Low voltageSag (momentary) · Brownout (prolonged)
No powerFault (momentary) · Blackout (prolonged)
Climate15–32°C · 20–80% humidity

The six power anomalies form a 3×2 grid the exam loves: high voltage — spike (momentary) and surge (prolonged); low voltage — sag/dip (momentary) and brownout (prolonged); total loss — fault (momentary) and blackout (prolonged). Defend in layers: surge protection and line conditioning for quality, UPS to bridge the gap, generator (tested under load, fuelled by contract) for the duration.

Climate numbers: temperature 15–32°C (59–89.6°F); humidity 20–80%. Too-high humidity corrodes components; too-low breeds static electricity (those winter shocks around the racks are a warning sign). Cooling vocabulary: sensible cooling removes heat; latent cooling removes moisture. HVAC for critical rooms is dedicated, monitored and positively pressurised — and in a fire it must automatically shut down (next panel).

Exam tip: Build the grid in your head: momentary/prolonged × high/low/none = spike/surge, sag/brownout, fault/blackout. “Voltage dropped for several minutes” → brownout.
3.9

Fire — Detection, Extinguisher Classes & Suppression Systems

Golden ruleLIFE SAFETY above everything
DetectorsIonization = earliest; photoelectric = smoke
Data centrePreaction water · FM-200 gas
ClassesA common · B liquid · C electrical · D metal

Every fire decision starts from the golden rule: life safety over assets, always. Detection by fire stage: ionization detectors sense the charged particles of the incipient stage — the earliest possible warning, before visible smoke; photoelectric detectors trigger when visible smoke scatters/blocks a light beam (stage 2); heat detectors are slowest but false-alarm-proof. The earlier the detection, the less damage from both the fire and its suppression. Extinguisher classes: A ordinary combustibles, B flammable/petroleum liquids, C electrical, D combustible metals (magnesium, sodium) — the wrong class makes it worse.

Water systems: wet pipe (always charged — fastest, leak risk); dry pipe (compressed air until triggered — for freezing climates); preaction (two-stage: fills on detection, releases only when a head melts — BEST for data centres because staff can abort a false alarm); deluge (all heads at once — never for electronics). Gas: FM-200 / Aero-K are the modern safe standards; Halon is banned (ozone depletion + health hazard); CO₂ is effective but lethal — unoccupied spaces with pre-discharge alarms only. And during any fire, HVAC shuts down automatically — otherwise it spreads smoke and feeds the fire oxygen.

Dig deeper: NFPA Codes & Standards

Exam tip: “Sprinklers that won't dump water on a false alarm” → preaction. “Earliest possible warning” → ionization. “Replace the Halon” → FM-200/Aero-K.

Sub-domain 3.10

Manage the information system lifecycle.

Security is a property of the whole lifecycle — from the first requirement to the last wiped disk. Bolting it on later is the most expensive mistake in engineering.

3.10

Security Across the System Lifecycle

PhasesRequirements → Design → Build → Operate → Retire
Cheapest fixAt REQUIREMENTS/design time
OperationsConfig & change management, patching
RetirementSanitise media — NIST SP 800-88

The lifecycle runs: stakeholder needs & requirements (security requirements defined alongside functional ones) → architecture & design (threat modeling, secure design principles from 3.1, evaluation criteria from 3.3) → development & implementation (secure coding, testing, certification) → integration & verification (does the built system meet the security requirements?) → operations & maintenance (configuration and change management, patching, continuous monitoring — and re-accreditation on significant change) → retirement/disposal.

Two lifecycle truths the exam rewards: defects cost orders of magnitude more to fix the later they're found — which is the entire argument for security-by-design; and systems are most vulnerable during transitions (upgrades, migrations, decommissioning). At end of life, media must be sanitised to NIST SP 800-88 (clear/purge/destroy — see Domain 2) or crypto-shredded where you can't touch the physical media (cloud). A retired system that still holds live data isn't retired — it's abandoned.

Dig deeper: NIST SP 800-160 — Systems Security Engineering · NIST SP 800-88 — Media Sanitization

Exam tip: Whenever a stem offers “add security review at requirements phase” against any later-phase fix, the earliest phase wins — cheapest, most effective, and ISC²'s philosophy in one answer.
